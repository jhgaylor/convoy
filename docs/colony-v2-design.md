# Design: Forge & Convoy v2 — the Colony Model

## One-line

Turn the game from "three identical drones each running a reflex" into **a slow,
ambient colony you program with a single brain, build out spatially, and grow
through committed time-and-cost investments** — while keeping everything good
about the current engine (deterministic sim, private rooms + one shared contested
market, convoys, credits-as-score, persistent world).

## The four decisions (this session)

1. **Control model: one colony brain.** The sim calls a single `tick()` per
   colony per tick. It sees everything you own + a world view, and returns a list
   of commands for all your units and buildings. (Screeps' `loop()`, not a
   per-unit reflex.)
2. **Depth system: spatial base-building.** You place buildings on your room grid
   (spawner, refinery, storage, fabricator, watchtower). Buildings cost goods and
   take ticks to construct. Units are spawned from buildings, cost resources, live
   under a population cap.
3. **Pacing engine: time + cost gating.** Long build/spawn times, a slow tick rate
   (~1/sec), tech tiers. You commit resources to a queue and watch it pay off.
   Cozy-strategic, checkable, the wait is the game.
4. **Migration: clean break to v2.** Replace `decide()` with the colony `tick()`
   ABI, rewrite the core loop in place, rewrite the example bots. No legacy ABI.
   (No real players to protect — the hosted instance is a public demo, no auth.)

## Problem statement

The current game is mechanically thin *to play*. The loop is solid and the engine
is genuinely good, but the player expresses strategy through a 14-int `decide`
function run independently per harvester. There's no coordination, no base to
command, no build-up. "How do you actually play" has no satisfying answer beyond
"tune a reflex and watch." We want more to do, a slower deliberate pace, and the
feeling of running a colony — buildings, unit types, a build-up that compounds.

## What makes this cool

You write **one brain** that runs your whole base. You watch a bare spawner and a
couple harvesters become a humming colony: ore flowing to a refinery, goods
stacking in storage, a build queue ticking down, convoys rolling out to the
contested market for credits. The fun is the same as Screeps and Factorio — *you
built the machine that plays the game* — but wrapped around the convoy/market
endgame that's already this project's best idea. Slow enough to be ambient: check
in, tweak the brain, watch it unfold.

## The new ABI (the load-bearing change)

Today (`lib/convoy/engine/wasm.ex`): `decide(14×i32) -> i32`, called per harvester.
Optional `convoy(9×i32) -> i32`.

v2: memory-based I/O, called once per colony per tick. Because the module has
**zero imports**, it cannot call back to the host to say "my commands are at
offset X" — so the contract below is explicit and pinned *now* (this is the
load-bearing artifact; everything inherits from it).

**The memory contract (frozen v1 — VALIDATED, see "Status" below):**

```
// Guest exports (besides memory):
inbuf()  -> i32               // address of a fixed static IN buffer; host writes the view here
tick(view_len: i32) -> i32    // returns command COUNT; writes records to the OUT buffer
outbuf() -> i32               // address of a fixed static OUT buffer; host reads commands here
```

Flow each tick: host reads `inbuf()` → writes the encoded view there → sets the
fuel budget → calls `tick(view_len)` → reads the returned count → reads `count`
fixed-stride records from `outbuf()`. The in-file SDK boilerplate hides all of
this; a bot author only edits the colony-logic function.

> **Note — this replaces an earlier `alloc()`/`out_ptr()` sketch.** A `no_std`
> single-file bot has no allocator, so a guest `alloc(len)` won't compile. Fixed
> static `IN`/`OUT` buffers exposed via `inbuf`/`outbuf` are the right shape, and
> a real compiled bot proved it (zero imports, exact exports). **Set fuel before
> any guest call** — even `inbuf`/`outbuf` trap on a zero budget.

**Wire format (frozen v1): fixed-size little-endian records, no varints.** A view
is a small header (counts) followed by fixed-stride arrays: units, buildings,
ore-deposits, queue entries, market intel. A command is a fixed 16-byte record
`(op:u8, _pad, target_id:u32, a:i32, b:i32)`. Fixed stride = trivial to decode
with no allocator and trivially bit-identical (determinism). TLV / forward-compat
is explicitly deferred — version the header and add fields at the end later.

- **View** (host → module): your units (id, type, pos, cargo), your buildings
  (id, type, pos, level, construction progress), your resources (ore/goods/credits,
  storage caps), the room's ore deposits, a build/spawn queue snapshot, and market
  intel (your convoys + nearest enemy convoy, as the `convoy` export gives today).
- **Commands** (module → host): written to a known out-buffer, resolved
  authoritatively by `Sim` in id order (intents-never-mutations holds):
  - `Move(unit_id, dx, dy)` — one greedy step, reusing the existing
    `World.step_toward`. **No A*/obstacle pathfinding** (doesn't exist today; not
    building it for v1). Buildings occupy cells; a step onto an occupied cell is
    blocked (no-op), same clamp-style resolution as today.
  - `Harvest(unit_id)`
  - `Transfer(unit_id, target_id)` — deposit ore into an *adjacent* refinery/storage
    (haulers); out of range = no-op.
  - `Build(building_type, x, y)` — queue construction at an empty cell
  - `Spawn(unit_type)` — from a spawner (costs goods + time)
  - `Upgrade(building_id)`
  - `LaunchConvoy(...)` — load goods, run to market (the endgame, preserved)
  - `Defend` / `Hunt(target)` — escort/convoy stances (today's `convoy` codes)
  - `Idle`
- **Invalid commands degrade to idle**, exactly like today's invalid intents. A
  command targeting a dead, nonexistent, not-yet-spawned, or enemy-owned id is a
  no-op. Stable ids come from the existing monotonic `World.next_entity_id`; the
  host hands those same ids in the view and never reuses them. The sim owns id
  validity — that's the anti-cheat surface.
- **Fuel** becomes ONE colony-wide budget per tick: the Region sets the fuel limit
  once (the existing `World.fuel_budget(world, player)` knob, applied to the single
  `tick` call instead of per-entity). Still the skill ceiling (primer §7), still
  instruction-counted and deterministic, tech-rewardable but capped. A hard
  command-count cap as a backstop on top of the fuel bound.
- **SDK matters now.** Hand-decoding a struct from bytes is miserable. Ship a
  `convoy-sdk` for Rust + AssemblyScript that exposes typed `World`/`Colony`/
  `Command` so a first bot is still ~20 lines. This is no longer optional polish;
  it's the onboarding. (Was already on the feature list.)

## Buildings (placed on the room grid, time + cost gated)

| Building | Glyph | Does | Notes |
|---|---|---|---|
| **Spawner** | 🏠 | produces units; sets population cap | the old `base` cell becomes this; cap scales with its level (Screeps GCL-like) |
| **Refinery** | ⚙ | ore → goods | **pulls refining out of the base** — today it's automatic/free; now it's a thing you build, place, and upgrade. Throughput = #refineries × level |
| **Storage** | 📦 | caps ore/goods you can hold | the resource-sink that forces "ship it or build more storage" decisions — a core pacing lever |
| **Fabricator** | ⚒ | goods → components → advanced units / convoy upgrades | the light production-chain / tech path (we picked base-building primary, but this is the upgrade ladder) |
| ~~**Watchtower**~~ | 🔫 | defense | **CUT from v1.** Introduces a whole new actor class (NPC bandits / raids) that isn't needed to prove the cozy-colony loop. Park it as a separate later feature; don't design around it now |

Construction: a `Build` command queues it; costs goods up front; takes
`build_time` ticks; (open question) requires a builder unit present, or auto-builds.
All times/costs are config knobs — extend the existing live admin config
(`World.@default_config`, already operator-tunable).

## Unit types (spawned from buildings, cost goods + time, pop cap)

| Unit | Does |
|---|---|
| **Harvester** ⛏ | mines ore (today's default behavior) |
| **Hauler** 🚛 | carries ore from nodes/harvesters to refinery/storage — logistics depth; lets harvesters keep mining (phase 3, can defer) |
| **Builder** 🔨 | constructs queued buildings — gates building behind a unit to manage |
| **Convoy/Escort** 🚚 | the market shipment unit + its defender — keeps the contested-market endgame intact, now produced like any unit |

## The economy spine (how it all connects)

```
ore (mined) → hauled to refinery → goods → spent on:
                                            ├─ spawn units (cost + time)
                                            ├─ build/upgrade buildings (cost + time)
                                            └─ load convoys → shared market → CREDITS (score)
```

Progression arc: spawner + 2 harvesters → build a refinery (goods flow) → build
storage (hold more) → add haulers (throughput) → fabricator (better convoys/
escorts) → ship convoys to the contested market for credits. **The market endgame
is preserved** — it's the payoff and the only PvP, exactly as primer §1 wants:
bases never attacked, the stake is the shipment in transit.

This keeps every good thing (determinism, private rooms + one shared contested
market, convoys, credits-as-score, persistent world, live config) and *adds* the
colony depth + one-brain control + slow pacing.

## Premises (agreed this session)

1. The `decide` ABI breaks; all example bots get rewritten. Accepted (clean break).
2. Fuel becomes one colony-wide budget per tick. Accepted.
3. **Warm fast-forward gets weaker, and we accept it.** Today refining is automatic
   (`World.refine_all/1`), which is *why* warm regions fast-forward (primer §5).
   Moving refine into a placed/queued building, and having a brain that issues
   commands every tick, means closed-form fast-forward really only holds for **cold**
   regions (no bot running) over command-free spans. An *active* colony must
   tick-simulate. The slow tick rate makes that cheap, so this is a fine trade — but
   it is a real reduction of the §5 property, stated plainly, not hidden.
4. This is engine v2 (World/Sim/Wasm/UI all change), phased. Accepted.

## Approaches considered (migration)

- **A — Sidecar prototype region** (`?mode=colony` alongside the live game). Lowest
  risk, prove-fun-first. *Not chosen.*
- **B — Clean break to v2** (rewrite the loop + ABI in place). **CHOSEN** — one
  coherent game, smallest conceptual surface, no legacy tax. Cost: a flag day where
  every existing bot must be rewritten (acceptable, no real players yet).
- **C — Dual ABI bridge** (keep `decide` + add `tick`). Gentlest migration, but two
  control models and double the test surface forever. *Not chosen.*

## Recommended build plan (phased, clean break)

- **Phase 0 — validate the (already-frozen) ABI from the author's seat.** The wire
  format + `alloc`/`tick`/`out_ptr` contract above is committed, so Phase 0 is no
  longer "go invent it" (that circular dependency with Phase 1 is removed). Phase 0
  is: build the `convoy-sdk` skeleton (Rust + AS) over the frozen format and write
  the simplest real colony bot against it — N harvesters mining to a refinery, queue
  a second refinery when affordable. If that's awkward to express, tweak the *record
  fields* (not the framing) before any Elixir.
- **Phase 1 — `tick` ABI + determinism.** Implement `alloc`/`tick`/`out_ptr` in
  `Wasm.ex` with the frozen layout; one colony-wide fuel limit per call; one unit
  type (harvester) driven by the brain, doing exactly what today's harvester does.
  Convoys stay on **auto-pilot** in Phase 1 (not yet brain-controlled) so the scope
  is just "prove the new control loop is bit-identical." Re-prove the §11 acceptance
  test: hot→cold→hot is bit-identical. This is the non-negotiable gate before any
  buildings.
- **Phase 2 — buildings + queue.** Spawner, refinery, storage; `Build`/`Spawn`
  commands; build queue with time+cost gating; refinery replaces auto-refine.
- **Phase 3 — logistics.** Haulers + `Transfer`; storage caps bite.
- **Phase 4 — endgame wiring.** Convoys/escorts as produced units into the existing
  contested market; fabricator + upgrades.
- **Phase 5 — surface.** UI: buildings on the grid, build-queue overlay,
  construction progress, colony panel (pop X/cap, queue). Rewrite example bots;
  polish the SDK. (The in-browser editor from the feature list pairs naturally here
  since the ABI is harder now.)

## Scope guard (what we are NOT doing)

- No territory expansion / multi-room claiming (rejected). One private room per
  player + one shared market stays.
- No upkeep/entropy treadmill (rejected pacing). Keep it cozy, not punishing.
- Combat stays at the market only; bases never attacked. No towers, no NPC
  bandits, no base defense in v1 — that's a separate feature for after the loop is fun.
- No A* pathfinding — greedy `step_toward` only; buildings just block cells.

## Open questions

- **Auto-build vs builder-unit-required?** Builder-required = more to manage and
  more readable pacing; auto = simpler v1. Lean builder-required, but could auto in
  Phase 2 and add the builder in Phase 3.
- ~~**View serialization format**~~ — RESOLVED: frozen fixed-stride little-endian
  records (see ABI section); TLV/forward-compat deferred behind a versioned header.
- **Command cap per tick** (fuel bounds it, but a hard cap is a sane backstop).
- **Cell occupancy rules:** can a unit stand on a building? can two buildings share a
  cell? (Lean: buildings block movement and each other; units pass over their own
  buildings. Pin in Phase 2.)
- **Pop-cap enforcement timing** — checked at `Spawn`-resolution against the
  accumulating world (so two spawns can't both exceed the cap in one tick).
- **Do haulers make v1 or defer?** Defer to Phase 3 — refinery-adjacent mining
  works without them at first.
- **Spawner count + pop-cap formula.** One spawner, cap = f(spawner level)? 

## Success criteria

- A new player writes a ~20-line colony bot (with the SDK) that mines, refines, and
  ships one convoy — and *enjoys* watching it grow over a few minutes.
- The determinism test suite (engine/wasm/compile) passes on the v2 loop: hot→cold→
  hot bit-identical (primer §11).
- The spectator page makes the build-up legible: you can watch a queue resolve and a
  colony fill in.

## Distribution

Web service (existing Phoenix deploy + sandboxed builder service — unchanged). The
genuinely new artifact is the **`convoy-sdk`** (Rust crate + AS package): ship it in
`examples/`/`docs/` first, consider crates.io / npm later. Without it, the harder ABI
tanks onboarding — treat the SDK as part of the feature, not a follow-up.

## The assignment (do this next, before any Elixir)

**Write the colony `tick()` for the simplest possible colony — N harvesters mining
to a refinery — as pseudocode against the proposed command list, from the bot
